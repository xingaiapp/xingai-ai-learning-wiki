# 重新设计理赔 Agentic 工作流:欺诈排序、升级回流路径与合规审计

一张流行的理赔自动化信息图把八个 Agent 排成一条直线——受理、文档核验、欺诈检测、损失评估、责任核对、审批、赔付、通知——中间穿插判断菱形,所有异常都汇入一个"人工复核与升级"框。乍看很顺,而且大方向*基本*是对的:真实理赔流程确实长得像一条流水线,带分级审批阈值和人工升级路径。但顺着三条线去追,就会露出把宣传图和真实系统设计区分开来的那些缺口:欺诈检测在拿到需要的数据之前就跑了;每条升级路径都倒进同一个框,却看不出怎么回到流程里;整张图里没有任何合规层——没有拒赔说理、没有审计轨迹、没有监管报送的挂钩点。

这篇文章重新设计这三处。不是要推翻原图的整体形状——多 Agent 流水线加分级审批阈值这个大结构是对的——而是修正那些在"这东西真要在生产环境里跑起来"这个问题下站不住脚的具体地方。

---

## 5W Framework

### What(是什么)

对一套通用理赔 Agentic 工作流做的三处结构性修正:

| | 原版 | 重新设计后 |
|---|---|---|
| 欺诈检测 | 一个"欺诈检测 Agent"排在损失评估之前 | 拆成**欺诈分诊**(早期、廉价信号:理赔频次、保单年限,排在损失评估*之前*)和**欺诈深度评分**(深层信号:成本异常、照片取证,排在损失评估*之后*) |
| 升级处理 | 所有异常路径都指向同一个"人工复核与升级"框,看不出明确的回流路径 | 一个**案件解决路由器**,携带明确的升级原因,把人工判断结果路由回*具体需要重跑*的那一步——而不是流程起点 |
| 合规 | 没有体现 | 一个横切全流程的**合规与审计轨迹 Agent**,每个环节都向它写入记录,支撑拒赔说理信、监管报送、公平性审计——复用 XingAI 内部已经采用过的同一套 Decision Ledger 形态 |

### Who(谁会用)

- 把理赔自动化的构想变成可落地设计的**企业/AI 架构师**
- 需要看清监管义务(拒赔说理通知、欺诈报送、及时给付计时器)到底挂在流水线哪个位置的**保险运营与合规相关方**
- 在这个工作空间里做下一个理赔 POC 的**工程师**,可以把这篇当作和 `claims-partner-api-mcp-poc`、`claims-mcp-oauth-poc` 并列的设计参考
- 正在审阅一张 AI 生成或供应商提供的理赔工作流图、需要一份"大概率漏了什么"检查清单的人

### When(什么时候用)

在开始实现之前,而不是之后。这里的三个缺口在纸面上改动成本很低,一旦代码已经写出来就贵得多:欺诈检测顺序在损失评估已经上线之后再调整,意味着要重新接两个 Agent 的输入管线;如果每条升级路径已经硬接到"重启整个理赔",之后再加路由器就等于要重新设计状态管理;而在一个已经在做拒赔决定的系统上补审计轨迹,意味着要为已经处理过的理赔回填根本不存在的合规证据。

### Where(用在哪)

具体针对保险理赔自动化,但这三种失败模式可以推广到任何"带欺诈/风险评分环节 + 人工升级路径 + 受监管或需可审计结果"的多 Agent 决策流水线:贷款审批、福利裁定、带申诉的内容审核、KYC/AML 审查。凡是信息图上出现"检测异常 → 评估 → 决策 → 升级给人工"这种结构,都值得对照检查这三点。

### Why(为什么重要)

因为这三个缺口失效的方式各不相同,而且没有一个会在 demo 里大声报警:

- **欺诈排序**是悄悄失效的——欺诈模型只是表现变差,该抓住的理赔(成本虚报是最常见的维修类理赔欺诈模式)在维修成本还不存在的时候算出来的欺诈分根本看不见,直到有人对某起欺诈案例做事后复盘、问"模型当时为什么没看出来"才会被发现。
- **升级路由**是运营层面失效的——每一次异常都变成整单重启,又慢又会让已经交过文档的客户被重复要求提供,而且"一起有争议的理赔实际要多久"这个问题根本回答不了,因为系统里没有"已解决,从第 6 步续跑"这种状态。
- **缺少合规**是法律层面失效的,而且只有在有人要求出示证据时才会暴露——很多司法辖区要求拒赔必须给出具体理由(不能是"AI 说不行"),而且这个理由要以理赔人能据此行动的形式给出;一家在自动化理赔决策时没有记录*为什么*这么决定的保险公司,事后根本拿不出这个理由,监管机构一旦问起模型是否系统性地不利于某个受保护群体,也没法证明。

## 修正一:欺诈检测得跑两次,不是一次

原设计把欺诈检测排在第三步,损失评估之前。这个顺序有个信息问题:相当一部分真实理赔欺诈——虚报维修估价、编造损失、照片对不上——只有在知道*成本*、拿到*照片*之后才能被发现,而这些在第三步都还不存在。

修法也不是简单地"把欺诈检测挪到损失评估后面",因为有些欺诈信号确实一开始就能拿到,不该等:理赔频次(同一份保单两个月内报了三次)、保单年限异常(出险发生在保单生效四天后)、理赔人/设备身份标记,这些评估根本不需要损失估价,而且早发现能省下一次对反正要被标记的理赔做完整损失评估的成本。

所以重新设计把欺诈检测拆成两个职责不同的 Agent:

- **欺诈分诊 Agent**(新增,排在第三步,损失评估之前):廉价的、基于身份和频次的信号。早期、低成本地拦下明显有问题的理赔。
- **欺诈深度评分 Agent**(新增,排在第五步,损失评估之后):需要损失评估输出才能算的深层信号——相对同类损失历史理赔的成本异常、照片取证/元数据分析,再和欺诈分诊已经标记的内容交叉核对。原图里那个单一的"欺诈分是否偏高?"判断真正应该放在这里,因为这是流水线里第一个模型真正拿到那个判断名字所暗示的数据的位置。

这不只是调整顺序——而是意识到"欺诈检测"其实是两份信息需求完全不同的工作,把它们塞进一步会逼出一个糟糕的取舍:要么跑得太早(判别性信号还不存在),要么跑得太晚(在本该更早叫停的理赔上浪费一次完整的损失评估)。

## 修正二:一个人工复核框不是设计,是待办事项

原图把所有异常——缺文档、欺诈分高、责任有争议、高额理赔——都路由进同一个"人工复核与升级"框,然后画一条虚线循环回"理赔提交"。这条回流路径对四种情况里的恰好一种说得通:客户补交缺失文档,确实需要重新从接近起点的地方进来。对另外三种情况,重启整个流水线要么是浪费(经理刚批准的高额理赔不需要重跑欺诈检测和损失评估),要么是明确错误(欺诈调查还了理赔人清白,不该把状态重置回受理,把已经确认的东西全部丢掉)。

重新设计用一个**案件解决路由器**替换了那单一个框:每条升级路径都携带一个标注好的原因(`missing_docs`、`fraud_investigation`、`estimate_dispute`、`high_value_review`),路由器把"原因 + 人工判断结果"映射到具体的重新接入点:

- `missing_docs` 已解决 → 回到**文档核验**(不是受理——理赔 ID 和已提取字段仍然有效)
- `fraud_investigation` 已排除 → 回到**责任核对**,欺诈标记被明确清除并记录这次清除
- `fraud_investigation` 已确认 → **拒赔**,案件关闭并报送反欺诈部门,不再循环到任何地方
- `estimate_dispute` 已调整 → 回到**审批**,带上理算员修订后的金额
- `high_value_review` 已批准 → 直接到**赔付处理**,跳过所有已经跑过、不需要再跑一遍的 Agent
- 任意路径,维持拒绝 → **拒赔**

重点不在于具体怎么映射——不同保险公司可能路由方式不一样——重点是*这套映射必须在设计里明确写出来、看得见*,而不是靠一条只对四种情况里一种说得通的虚线暗示出来。"人工复核与升级"本身是个真实且必要的组件;只是它自己回答不了"之后怎么办"这个完整问题。

## 修正三:没有合规层就没有证据

原图里没有任何东西是为以后留存的。审批 Agent 有一个"决定"输出,赔付处理有一个"结算记录"输出,但没有任何东西以理赔人或监管机构能用的形式记录*为什么*会拒赔,也没有任何东西记录是哪个模型版本做的这次判断。

重新设计加了一个明确横切全流程的**合规与审计轨迹 Agent**——它不是一个带进出箭头的流水线步骤,而是一条其他每个环节都要往里写的车道。它用的形态和 XingAI 好几个产品已经采用的 Decision Ledger 模式(`xingai-engineering-system/patterns/decision-ledger-schema.md`,最近一次是 `xingai-learn` 的 ADR-003)完全一致:每个决策一行,包含 `domain`、`recommendation`、`reasoning`、`confidence`,以及——对这个场景特别关键的——指回具体是哪个模型版本、引用了哪条保单条款做出这次判断的指针。

这份台账支撑着原设计完全没有产出路径的三件事:

- **拒赔说理信**,从责任核对 Agent 引用的具体保单条款直接生成——不是一句笼统的"理赔被拒绝"。很多司法辖区的不公平理赔行为法规要求这个具体程度,而如果决策当时没有记录推理过程,事后根本没法重建。
- **监管报送**:很多司法辖区的及时给付法要求在一份清晰理赔提交后 N 天内做出决定;欺诈已确认的案件通常带有向州保险监管部门或反欺诈局报送的义务。两者都需要每个环节何时运行的带时间戳证据,而这只有在有东西一直在记录时才存在。
- 对两个风险最高的自动化决策——**欺诈深度评分**和**审批**——做**公平性/偏见审计**,每次决策都锁定对应的模型版本,这样一个被标记的结果能追溯到具体是哪个模型做出的,而不是"上个季度某个时间点在线的某个版本"。

这里也是赔付处理的幂等性要求该被放进同一个讨论的地方:第八步是真金白银在动,这个工作空间里的 `claims-partner-api-mcp-poc` 已经把赔付写操作的幂等键当作不可妥协的东西来处理,理由和审计轨迹在这里不可妥协是一样的——两者都是关于事后能不能回答"实际发生了什么",而不只是设计上"应该发生什么"。

## 哪些地方原封不动地保留了下来

值得明确说一句,免得这篇重新设计的文章读起来像"原图全错了":原图的核心形状是站得住的。多渠道受理、每个环节都有明确结构化输出的顺序专职 Agent、按理赔金额分级的审批阈值,以及*某种*人工介入的升级路径,这些判断都是对的,也确实体现了真实的业务领域认知——这不是一个"AI 噱头图全错了"的案例,而是一个大方向正确的架构,里面有三处具体地方经不起"这东西要连续跑一年、还得应付监管机构提问"这个问题的推敲。
