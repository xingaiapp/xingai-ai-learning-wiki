# Agent 治理参考架构：授权、溯源、审批、审计

每一场企业 Agent 讨论最终都会到达同一个问题，通常由会议室里最资深的人问出：

> *这个 Agent 不经任何人批准到底能做什么——它做了之后，记录在哪里？*

多数团队用模型层的保证来回答（"system prompt 里禁止了"）。这个答案过不了评审，也不该过：Mozilla 0din 团队展示了一个外观干净的 GitHub 仓库可以引导 coding agent 执行恶意命令。注入是通过 Agent *本来就该读*的内容进来的。写在 prompt 里的规则，约束不了一个输入流可被攻击者触达的 Agent。

Microsoft 的开发者研究（《To Copilot and Beyond》）指出了真正答案的形状：人们对 Agent 真正想要的是**有边界的委托**（bounded delegation）——在明确、可检视的边界内的真实自治。治理就是让边界成真的那套系统。

本文给这套系统一个参考架构：五个平面，每个都有可检视的具体实现。这些不是幻灯片架构——主参考实现（`xingai-agent-firewall`）今天就在跑，且该模式已以领域专用形式落地三次（交易门禁、预测市场门禁、理赔裁定升级）。

---

## 核心倒置

传统安全问的是*用户是谁、可以访问什么*（authN/authZ）。Agent 治理问的是另一个问题：

> **委托人已经被授权了。Agent 在没人能完全控制的指令之下，可以代表他做什么？**

这个倒置正是 IAM、DLP、网络策略覆盖不了它的原因：Agent 在用户会话*内部*运行、持有用户的权限，但它的行为是它读过的每个文档、仓库、网页的函数。威胁模型不是恶意的外部人，而是**以机器速度运转的、被诱导的内部人**。

## 五个治理平面

```
        Agent 意图（工具调用 / 下单 / 写入 / 发消息）
                          │
  ┌───────────────────────▼───────────────────────────┐
  │ 1. 授权平面            什么不需审批、什么需要、      │
  │    （声明范围）        什么永远不可                  │
  ├────────────────────────────────────────────────────┤
  │ 2. 策略平面            确定性规则 + 加权信号 →       │
  │    （门禁本体）        allow | deny | review        │
  ├────────────────────────────────────────────────────┤
  │ 3. 审批平面            review 同步挂起；             │
  │    （human-in-loop）   超时即拒绝                    │
  ├────────────────────────────────────────────────────┤
  │ 4. 溯源平面            这条指令从哪来？可信配置      │
  │    （来源追踪）        还是刚抓取的内容               │
  ├────────────────────────────────────────────────────┤
  │ 5. 审计平面            每个裁决一行 Decision，       │
  │    （台账）            含人工推翻                    │
  └────────────────────────┬───────────────────────────┘
                           ▼
              效果执行（或永不执行）
```

### 1. 授权平面 —— 把有边界的委托写下来

每个 Agent 界面声明三张清单：**自由执行**（读、搜、起草）、**需要审批**（执行、工作区外写入、花钱）、**永远不可**（外传凭证、绕过自己的门禁）。这份文档正是企业买家索要的东西，同时兼任测试规格：我们理赔裁定 POC 的验收套件最重点测试的就是一条性质——*超过升级阈值的理赔永远不被自动裁定*。

### 2. 策略平面 —— 裁判必须不可被 prompt

allow/deny 由确定性规则产生：具名加权信号、配置化阈值、相同输入 → 相同裁决、每条触发规则都列出。防火墙实现里：`shell_pipe_from_network` +40、`secrets_path_access` +35，共七个信号，YAML 配置。

诱人的替代方案——问 LLM"这危险吗？"——只是把漏洞往下挪了一层：一个会阅读攻击者可控内容的裁判本身就可被注入。LLM 可以贡献*参考*信号提高审查等级，但永远不能授权规则已拒绝的行为。

### 3. 审批平面 —— 同步、fail-closed

`review` 裁决把动作挂起等人决定，四种归宿：批准一次、本 session 批准、拒绝、拒绝并加规则。拒绝并加规则是两层，不是一次写入（[Agent Firewall ADR-005](https://github.com/xingaiapp/xingai-agent-firewall/blob/main/docs/adr/005-deny-add-rule.zh.md)）：即时可逆的 **pinned deny** 阻止同一调用再进队列；**rule suggestion** 把耐久 YAML 改动排进人工评审。运行时绝不自己改写 `policies/*.yaml`——强制 Agent 行为的配置，仍像代码一样被评审。

两个默认不可协商：**超时即拒绝**、**门禁不可达即阻断**。fail-closed 必须贯穿全链路——一个会 fail-open 的审批系统只是穿着戏服的审计系统。

异步审批（先失败、批准后重试）听起来更干净，实际更糟：Agent 会把失败当错误并即兴绕路——这恰恰是治理层最不能诱发的行为。

第二个实现，[`xingai-robinhood-mcp` ADR-001](https://github.com/xingaiapp/xingai-robinhood-mcp/blob/main/docs/adr/001-mcp-gateway-proxy.zh.md)，把拦截点本身挪了位置：不是 harness hook，而是客户端连接的一个本地 **MCP 网关代理**，代替直连第三方 MCP 端点——一旦某个工具要从好几个 harness（Cursor、Claude Code、ChatGPT）里用，这一步就是必须的，因为它们没有共享的 hook 契约。它还把 fail-closed 的纪律又收紧了一层：Invest AI ADR-028 的七道交易门禁里，三道（人工确认、账户范围检查、审计日志）在这里是真代码；另外四道没有被桩成"通过"——每一道都点名拒绝（"G3 数据新鲜度未接入"），直到它需要的跨仓库依赖真正存在为止。一个还没能力强制某道门禁的治理层，必须逐门声明，而不是把它平均掉、伪装成一份完整的清单。

### 4. 溯源平面 —— 信任是来源的属性

0din 攻击成立，是因为刚 clone 的 README 里的指令和用户亲手输入的请求权重相同。治理要求相反：可溯源到不可信来源（新仓库、抓取页面、外来文档）的指令，会提高其触发的任何动作的风险分。

这个平面在防火墙参考实现里已经**落地**（[ADR-004](https://github.com/xingaiapp/xingai-agent-firewall/blob/main/docs/adr/004-origin-provenance-tracking.zh.md)），不是愿景。机制：session 启动时 `git ls-files` 基线；PostToolUse 对 Read/Grep/Glob/WebFetch/WebSearch 打污染；UserPromptSubmit 清零（按轮次，不是时间窗）。引擎从 session 污染状态推导 `untrusted_origin_instruction`——PreToolUse hook 保持薄。同样的 0din 复现，在本轮读过恶意 README 之后，从风险分 60（`review`）升到 80（`deny`）。

诚实边界仍在：非 git 项目只有网页溯源；跨轮延迟攻击是迁移触发条件，不是 v1 覆盖面。溯源信号单独 fail-open——缺基线不发明假拒绝；其他策略信号照样工作。

### 5. 审计平面 —— 一个 schema，每个裁决

每个受门禁动作写一行决策：尝试了什么、触发了哪些规则、裁决是什么、谁推翻了、实际执行了什么。两个设计选择很关键：

- **人工推翻是一等数据。** 人批准了规则标记的（或拦下了规则放行的），是调校权重的最高价值信号——按规则展示推翻率。频繁被钉死的模式（ADR-005）是同一种信号的另一种形态：该写 YAML 规则了。
- **跨产品共享一个 schema。** XingAI 的 Decision Ledger schema 如今在交易、餐食、学习、研究、Agent 安全领域有六个采用者。跨产品审计视图零额外成本，因为形状从第一天就是共享的。

## 参考实现

| 平面 | 通用实现 | 领域专用先例 |
|---|---|---|
| 授权 | 防火墙威胁模型 + 受管控类别配置 | Invest AI ADR-028（G1–G7 交易门禁）、理赔 POC 升级阈值 |
| 策略 | YAML 信号引擎（确定性） | 欺诈检查"规则优先、LLM 其次" |
| 审批 | 审批队列、超时即拒绝、pin + 建议（ADR-005） | 交易确认弹窗、Telegram 确认、Robinhood MCP 网关 G1（`xingai-robinhood-mcp` ADR-001） |
| 溯源 | 按轮次 session 污染 + 基线（ADR-004） | 引用锚定 RAG（无引用文本到不了裁定者面前） |
| 审计 | Decision Ledger schema | audit_trail 表、决策快照、Robinhood MCP 网关 G7（`xingai-robinhood-mcp` ADR-001） |

该模式已在 XingAI 工程体系中固化为 `agent-execution-gate`，附按产品的采用 checklist——新执行面的采用方式是写一页 ADR，把五个平面映射到自己身上。

## 要避免的

- **只靠 prompt 治理** —— 那是请求不是门禁；注入直接走过去。
- **一个中心化门禁服务管所有产品** —— 每个产品用自己的信号守自己的面；共享的是模式和台账 schema，不是运行时依赖。
- **只审计不阻断** —— 到审计时载荷已经跑完了。
- **裁决路径上的 LLM 裁判** —— 只做参考、限权、永不授权。
- **运行时改写策略 YAML** —— pin 是运行时状态；耐久规则走 git 评审。
- **跳过诚实的威胁模型** —— 每个门禁都必须写清它*不能*挡什么（防火墙挡被诱导的 Agent，挡不了有 settings 权限的恶意本机用户）。宣称完备的治理叙事过不了尽调；写明边界的才过得了。

## 采用路径

1. **第 1 周：** 为一个 Agent 界面声明授权范围；接一个 Agent 绕不过去的拦截点（harness hook、网关、或门禁函数）；上线确定性策略 + fail-closed 检查。
2. **第 2 周：** 超时即拒绝的审批流；若界面会读不可信内容（仓库、网页、外来文档），接上按轮次溯源。
3. **第 3 周：** 台账行 + 推翻率视图；耐久策略用 pin + 建议（不现场改 YAML）；回归语料进 CI（已知恶意永不静默放行、已知良性永不误拦）。
4. **之后：** 团队级策略继承、集群看板、多 harness 适配——在单界面闭环端到端跑通*之后*。

Agent 会越来越强。企业的问题已经在从"你的 Agent 多能干"转向 **"它不经请示能做什么、剩下的谁批、记录在哪"**。能用架构图和一条台账查询——而不是一段 system prompt——回答这个问题的团队，才是拿到部署许可的团队。
